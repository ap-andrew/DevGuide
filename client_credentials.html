In [node:3972], you configured API key validation for your API. In this quick start, you configure policies similar to the [node:3972], but instead of the API key pattern, the policies enforce OAuth 2.0 client credentials.

Client credentials is a good grant type to use for initial OAuth setup, since, unlike the other grant types defined in OAuth 2.0, client credentials does not entail redirecting the app end user to a login page. Therefore, the complexity of setup is minimized. A consequence of this simplicity is that client credentials is only margnially more effective than API key validation. For this reason, in real-world applications, the client credentials grant type is usually only used in scenarios in which the apps that are being authorized are trusted clients, such as those developed and distributed by the API providers themselves or by close partners.

Step 1: Create policies

As in [node:3972], you create a set of policies in your API proxy under the /policies directory. (The ClassifyRequest policy and CheckQuota policy are the same in this quick start--no changes are required.)

ClassifyRequest policy of type AssignMessage

In apiproxy/policies, create a file called ClassifyRequest.xmlwith the following content:

<AssignMessage name="ClassifyRequest"> 
  <AssignVariable> 
    <Name>flow.resource.name</Name> 
    <Ref>proxy.pathsuffix</Ref> 
  </AssignVariable> 
</AssignMessage>

CheckQuota policy of type Quota

In apiproxy/policies, create a file called CheckQuota.xml with the following content:

<Quota name="CheckQuota"> 
  <Interval ref="apiproduct.developer.quota.interval"/>
  <TimeUnit ref="apiproduct.developer.quota.timeunit"/>
  <Allow countRef="apiproduct.developer.quota.limit"/>
  <Identifier ref="client_id"/>
</Quota>

Return404 policy of type RaiseFault

This policy returns an HTTP 404 error if the API proxy cannot classify a request.

In apiproxy/policies, create a file called Return404.xml with the following content:

<RaiseFault name="Return404">
  <FaultResponse>
    <Set>
      <StatusCode>404</StatusCode>
      <ReasonPhrase>Not Found</ReasonPhrase>
      <Payload contentType="text/plain">Not Found</Payload>
    </Set>
  </FaultResponse>
</RaiseFault>

ValidateOAuth policy of type OAuthV2

This policy verifies that the request contains a proper OAuth 2.0 bearer token, validates the API product, and authorizes access to the requested API resource.

In apiproxy/policies, create a file called ValidateOAuth.xml with the following content:

<OAuthV2 name="ValidateOAuth"> 
  <Operation>VerifyAccessToken</Operation> 
</OAuthV2>

Set up a token endpoint to distribute OAuth 2.0 access tokens

In apiproxy/policies, create a file called GenerateAccessToken.xmlwith the following content:

<OAuthV2 name="GenerateAccessToken">
  <!-- Generates an OAuth 2.0 access token using the client_credentials grant type -->
  <Operation>GenerateAccessToken</Operation>
  <!-- This is in millseconds, so expire in an hour -->
  <ExpiresIn>3600000</ExpiresIn>
  <SupportedGrantTypes>
    <GrantType>client_credentials</GrantType>
  </SupportedGrantTypes>
    <GrantType>request.queryparam.grant_type</GrantType>
  <GenerateResponse/>
</OAuthV2>

Step 2: Attach policies to proxy endpoint

Attach these policies to the proxy endpoint (/proxies/default.xml). The structure of this proxy endpoint includes a request preflow step, which will execute for every request received by this API proxy. The request preflow enforces the ClassifyRequest policy, which sets a variable, the URI path suffix. The URI path suffix is used by the processing pipeline to determine which flow to execute.

Three named flow are defined: GetAccessToken, Weather, and Fallthrough. The first two flows define a condition for execution.

  1. If the path suffix of the request is /accesstoken, then execute the GetAccessToken flow.
  2. If the path suffix of the request is /weather, then execute the Weather flow.
  3. If neither condition is satisfied, then execute the 'fallthrough' flow, and reject the request with a 404 error.

Configure this flow structure in /proxies/default.xml as follows:

<ProxyEndpoint name="default">
  <PreFlow>
    <!-- Executes for all inbound requests. Sets the URI path suffix -->
    <Request>
      <Step><Name>ClassifyRequest</Name></Step>
    </Request>
  </PreFlow>
  <Flows>
    <Flow name="TokenEndpoint">
    <!-- Executes when the request URI path suffix is /accesstoken -->
      <Condition>proxy.pathsuffix MatchesPath "/accesstoken"</Condition>
      <Request>
        <Step><Name>GenerateAccessToken</Name></Step>
      </Request>
    </Flow>
    <Flow name="Authorize">
    <!-- This policy flow is selected for all other API calls -->
      <Request>
        <Step><Name>ValidateOAuth</Name></Step>
        <Step><Name>CheckQuota</Name></Step>
      </Request>
    </Flow>
    <Flow name="Fallthrough">
    <!-- Return 404 for invalid URLs -->
      <Request>
        <Step><Name>Return404</Name></Step>
      </Request>
    </Flow>
  </Flows>
  <HTTPProxyConnection>
    <BasePath>/weather</BasePath>
    <VirtualHost>default</VirtualHost>
  </HTTPProxyConnection>
  <RouteRule name="default">
    <TargetEndpoint>default</TargetEndpoint>
  </RouteRule>
</ProxyEndpoint>

Step 4: Obtain access token and invoke the API

Obtain an access token by contacting the token endpoint with valid credentials. Since you configured the client credentials grant type, an app can obtain an access token using only a valid consumer key and secret pair--no redirects are involved.

As an admin, you can always obtain the consumer key and secret for the app you created in [node:427], by retrieving the app profile:

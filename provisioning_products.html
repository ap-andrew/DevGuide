This topic covers the definition of API products. API products enable API providers to create bundles of URIs that are combined with Quota settings. The goal of API products is to enable API providers to expose a single set of RESTful URIs as multiple different bundles, each of which is associated with a 'service plan'. This enables API providers to create bundles on-the-fly, without requiring recoding of backend services. In a typical scenario, a set of URIs are bundled into two API products, 'Free' and 'Premium', where 'Free' provides a low quota, and Premium provides a higher quota at a cost. Access to API products is enforced by OAuth. The following Developer Guide topics provide instruction on setting up OAuth.

Defining API products

An API product is a list of API resources (URIs) and rate-limiting (quota) settings. Thus to define an API product you must decide which URIs to bundle, and how many requests should be permitted over a given time interval, for example, 1 request per minute, or 10,000 requests per month. The service plan that you create applies to all URIs in the bundle.

API products expose the following configuration options:

  • apiResources: A comma separated lists of URIs bundled into the API product. Wildcards are supported. In the API product below,  the API resource is "/forecastrss", which means that only requests to/weather/forecastrss are permitted for apps that are approved for this API product. You can define several API products to "bundle" API resources for different business or technical purposes.
  • approvalType: Automatic or Manual approval for consumer keys returned to apps enrolled for this product. If set to manual, the consumer key is returned in the 'pending' state, and must manually be approved before the app can access the provider resources.
  • attributes: A set of optional, custom attributes that may be to extend the default API product profile
  • description: A description of the API product (which will be exposed to developers via a developer portal, for example)
  • scopes: An optional, comma-separated list of OAuth scopes that are validated at runtime. (The API Platform validates that the scopes in any access token presented match the scope set in the API product)
  • proxies: Named API proxies. By specifying proxies, you can bind the API resources (URIs) listed in the API product to specific API proxies, preventing developers from accessing those URIs via other API proxies. Note that if you do not configure proxies or environments, you must set the variable proxy.pathsuffix explicitly, using an ExtractVariables policy.
  • environments: Named environments (for example 'test' or 'prod"). By specifying one or more environment, you can bind the API resources listed in the API product to specific environment, preventing developer from accessing those URIs via API proxies in another environment. This setting is used, for example, to prevent URIs protected by API proxies in 'prod' from being accessed by API proxies deployed in 'test'. Note that if you do not configure proxies or environments, you must set the variable proxy.pathsuffix explicitly, using an ExtractVariables policy.
  • quota: Number of requests permitted per app over the time interval specified.
  • quotaInterval: Number of time units over which quotas are evaluated.
  • quotaTimeUnit: The time unit (minute, hour, day, or month) over which quotas are counted.

The API Platform exposes an API that enables API product management. An API product can be created with a POST request, as in this example:

$ curl -H "Content-Type:application/json" -X POST -d \
'{
  "apiResources" : [ "/forecastrss" ],
  "approvalType" : "auto",
  "attributes" : [ ],
  "description" : "A bundle of API resources (URIs) that can be access for free",
  "displayName" : "Free API bundle",
  "environments" : [ "test" ],
  "name" : "FreeAPIProduct",
  "proxies" : [ "weatherapi" ],
  "quota" : "1",
  "quotaInterval" : "1",
  "quotaTimeUnit" : "minute",
  "scopes" : []
}' \
-u myname:mypass https://api.enterprise.apigee.com/v1/o/{org_name}/apiproducts

Sample Response

{
  "apiResources" : [ "/forecastrss" ],
  "approvalType" : "auto",
  "attributes" : [ ],
  "createdAt" : 1361457299970,
  "createdBy" : "admin@#apifactory.com",
  "description" : "A bundle of API resources (URIs) that can be access for free",
  "displayName" : "Free API bundle",
  "environments" : [ "test" ],
  "lastModifiedAt" : 1361457299970,
  "lastModifiedBy" : "admin@apifactory.com",
  "name" : "FreeAPIProduct",
  "proxies" : [ "weatherapi" ],
  "quota" : "1",
  "quotaInterval" : "1",
  "quotaTimeUnit" : "minute",
  "scopes" : [ ]
}

The API product above authorizes access to the '/forecastrss' URI. Once appropriate OAuth policies are configured, any request with a valid credential made to the URI path following /weather/forecastrss will be authorized. (Note that request path consists of the base path (configured in the ProxyEndpoint) and URI suffix, which is defined as the API resource.)

The API product above defines settings for the API product's 'quota'. Developers who sign up for this API product will be able to build apps that consume the API resource /forecastrss under the conditions of the quota settings, that is, 1 request per minute.

Creating developer profiles

Developers are registered in an organization by creating a profile. This step typically occurs through a developer portal, and, indeed, this API is the same API that is called by Apigee API Platform Developer portal. Note that the developer email that is included here is used as the unique key for the developer throughout the system. Arbitrary attributes can be added to the developer profile as needed. The attributes are not interpreted by the API Platform, but can be of use in custom analytics, custom policy enforcement, and so on.

$ curl -H "Content-type:application/json" -X POST -d \
      '{"email" : "ntesla@theramin.com", 
        "firstName" : "Nikola", 
        "lastName" : "Tesla", 
        "userName" : "theramin", 
        "attributes" : [ { "name" : "project_type", "value" : "public"} ] }' \
-u myname:mypass https://api.enterprise.apigee.com/v1/o/{org_name}/developers

Sample Response

{
	  "email" : "ntesla@theramin.com",
	  "firstName" : "Nikola",
	  "lastName" : "Tesla",
	  "userName" : "theramin",
	  "organizationName" : "{org_name}",
	  "status" : "active",
	  "attributes" : [ {
	    "name" : "project_type",
	    "value" : "public"
	  } ],
	  "createdAt" : 1343189787717,
	  "createdBy" : "admin@apigee.com",
	  "lastModifiedAt" : 1343189787717,
	  "lastModifiedBy" : "admin@apigee.com"
	}

Registering developer apps

Every app that is created on the API Platform is associated with a developer and an API product. When an app is registered on behalf of a developer, the API Platform generates a "credential" (a consumer key and secret pair) that identifies the app. This credential is 'bound' to at least one API product. The credential generated for an app enables it to consume a set of URI paths at a specified rate (defined by the quota settings in the API product).

$ curl -H "Content-type:application/json" -X POST -d \
      '{"apiProducts": [ "weather_free"], 
        "callbackUrl" : "login.weatherapp.com", 
        "name" : "weatherapp"}' \
-u myname:mypass https://api.enterprise.apigee.com/v1/o/{org_name}/developers/ntesla@theramin.com/apps

Sample Response

{
  "callbackUrl" : "login.weatherapp.com",
  "createdAt" : 1343190620890,
  "createdBy" : "admin@apigee.com",
  "credentials" : [ {
    "apiProducts" : [ {
      "apiproduct" : "weather_free",
      "status" : "approved"
    } ],
    "attributes" : [ ],
    "consumerKey" : "HQg0nCZ54adKobpqEJaE8FefGkdKFc2J",
    "consumerSecret" : "1eluIIdWG3JGDjE0",
    "status" : "approved"
  } ],
  "lastModifiedAt" : 1343190620890,
  "lastModifiedBy" : "admin@apigee.com",
  "name" : "weatherapp",
  "status" : "approved"
}

Note that for collective ownership. apps can also be associated with company entities. Such apps are referred to as 'company apps' (as opposed to 'developer apps').

To manage company apps, you can use the /companies/{company_name}/apps API resource.

Managing consumer keys for apps

Get the consumer key (the API Key) for the app

The credentials for an app, (API product, consumer key and secret) are returned as part of the app profile. An administrator of an organization can retrieve the consumer key at any time, for developer support, for example.

Every consumer key has a profile, which an administrators in an organization can retrieve. The profile shows the status of the consumer key, as well as any API product associations for the key.

$ curl -H "Accept: application/json" \
-u myname:mypass https://api.enterprise.apigee.com/v1/o/{org_name}/developers/ntesla@theramin.com/apps/weatherapp/keys/HQg0nCZ54adKobpqEJaE8FefGkdKFc2J

Sample Response

{
  "apiProducts" : [ {
    "apiproduct" : "weather_free",
    "status" : "approved"
  } ],
  "attributes" : [ ],
  "consumerKey" : "HQg0nCZ54adKobpqEJaE8FefGkdKFc2J",
  "consumerSecret" : "1eluIIdWG3JGDjE0",
  "status" : "approved"
}

Approving consumer keys

Consumer keys must be explicitly approved when API products have key approval set to manual. Keys can be explicitly approved using this method call:

$ curl -u myname:mypass https://api.enterprise.apigee.com/v1/o/{org_name}/developers/ntesla@theramin.com/apps/weatherapp/keys/HQg0nCZ54adKobpqEJaE8FefGkdKFc2J?action=approve

Sample Response

{
	  "apiProducts" : [ {
	    "apiproduct" : "weather_free",
	    "status" : "approved"
	  } ],
	  "attributes" : [ ],
	  "consumerKey" : "HQg0nCZ54adKobpqEJaE8FefGkdKFc2J",
	  "consumerSecret" : "1eluIIdWG3JGDjE0",
	  "status" : "approved"
	}

Approving API products for consumer keys

The association of an API product with a key also has a status. A consumer key can be 'approved' in general, but not approved specifically for an API product. For API access to be successful, the consumer key must be approved, and the consumer key must be approved for the proper API product. The association of a consumer key with an API product can be approved as follows:

$ curl -u myname:mypass https://api.enterprise.apigee.com/v1/o/{org_name}/developers/ntesla@theramin.com/apps/weatherapp/keys/HQg0nCZ54adKobpqEJaE8FefGkdKFc2J/apiproducts/weather_free?"action=approve""

To enforce API product access at runtime, OAuth and quota policies must be configured. To learn how to configure runtime enforcement of API products, see [node:3972]
